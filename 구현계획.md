# 하이큐 배구게임 구현계획

---

## 1. 아키텍처 변경 개요

### 기존 구조 (v0.1.0)
- `game.start(match, { initState, tick })` — 게임로직이 tick 함수를 직접 구현
- StateSystem이 tick 콜백을 주입받아 호출
- 스프라이트 인덱스가 EntityManager에 고정

### 새 구조
- tick 함수가 **엔진(GameLoop.js)** 안에 존재
- 게임로직은 **이벤트 핸들러**와 **엔티티 설정**만 제공
- 엔티티 설정(액션, 스프라이트, 히트박스)이 EntityManager에 저장
- StateSystem은 순수 상태 컨테이너로 축소

```
GameBuilder
  └─ Game (internal)
       ├─ GameLoop        ← 신규 (엔진 내부)
       │    참조: EntityManager, PhysicsMap, handlers
       ├─ StateSystem     ← 축소 (순수 상태 컨테이너)
       ├─ Renderer        ← spriteIndex 계산 방식 변경
       ├─ Effector
       └─ InputSystem
```

## 3. 엔티티 설정 스키마 (EntityManager 저장 형식)

빌드 시 `entityManager.register(id, entity)`에 전달하는 entity 객체의 전체 스키마.

```js
entityManager.register('player1', {
  // ─── 렌더링 ─────────────────────────────────────────
  type: 'entity',        // 'bg' | 'entity' | 'ui'
  assetId: 'p1_sheet',  // AssetStore 에셋 이름

  // ─── 물리 설정 ───────────────────────────────────────
  physics: {
    gravity: 0.0005,       // 매 틱 vy 감소량 (physics unit)
    restitution: 0,        // 0 = 비탄성
    playerSide: 'left',    // 'left' | 'right' | null (공, 네트는 null)
  },

  // ─── 액션 설정 ───────────────────────────────────────
  // 각 액션은 duration, 스프라이트 범위, 히트박스 생성 함수를 가짐
  actions: {
    IDLE: {
      duration: 0,                        // 0 = 입력으로만 종료
      sprites: { start: 0, count: 3 },   // 스프라이트 시트 프레임 범위
      getHitbox: (t, facing) => [ ... ], // t = actionTick/actionDuration (0~1)
    },
    JUMP: {
      duration: 0,
      sprites: { start: 3, count: 4 },
      getHitbox: (t, facing) => [ ... ],
    },
    SPIKE: {
      duration: 10,
      sprites: { start: 7, count: 6 },
      getHitbox: (t, facing) => [ ... ], // t에 따라 팔 캡슐 각도 보간
    },
    BLOCK:   { duration: 15, sprites: { ... }, getHitbox: (t, facing) => [...] },
    DIVE:    { duration: 35, sprites: { ... }, getHitbox: (t, facing) => [...] },
    RECEIVE: { duration: 15, sprites: { ... }, getHitbox: (t, facing) => [...] },
    RUN:     { duration: 0,  sprites: { ... }, getHitbox: (t, facing) => [...] },
    SKILL:   { duration: 30, sprites: { ... }, getHitbox: (t, facing) => [...] },
  },
});

// 공 엔티티
entityManager.register('ball', {
  type: 'entity',
  assetId: 'ball_img',
  physics: { gravity: 0.0003, restitution: 0.9, playerSide: null },
  actions: {
    DEFAULT: { duration: 0, sprites: { start: 0, count: 1 }, getHitbox: () => [...] },
  },
});
```

**getHitbox(t, facing)** 반환값:
```js
// Physics 바디 배열 (Physics.js 스키마 준수, wx/wy 없는 오프셋 기준)
[
  { shape: 'circle',  ox: 0, oy: -0.070, r: 0.0175, restitution: 0 },
  { shape: 'capsule', ox: 0, oy: -0.040, length: 0.050, angle: Math.PI/2, r: 0.020, restitution: 0 },
  // SPIKE일 때 t에 따라 팔 캡슐 각도가 lerp됨
]
```

## 4. 게임로직 인터페이스 (handlers)

`game.start(config)`에 전달하는 객체.

```js
game.start({
  physicsMap: new PhysicsMap(1, 1.125),  // 물리맵 크기

  // 엔티티 초기화 (EntityManager 등록 + 초기 상태 반환)
  initEntities(entityManager) {
    entityManager.register('player1', { ... });
    entityManager.register('player2', { ... });
    entityManager.register('ball', { ... });

    return {
      // 물리 상태 (physics unit)
      player1: { x: 0.2, y: 0, vx: 0, vy: 0, facing: 1, onGround: true,
                 actionType: 'IDLE', actionTick: 0, stamina: 100 },
      player2: { x: 0.8, y: 0, vx: 0, vy: 0, facing: -1, onGround: true,
                 actionType: 'IDLE', actionTick: 0, stamina: 100 },
      ball:    { x: 0.5, y: 0.3, vx: 0.002, vy: 0 },

      // 게임 규칙 상태
      phase: 'serve',
      score: { p1: 0, p2: 0 },
      sets:  { p1: 0, p2: 0 },
    };
  },

  handlers: {
    // 액션 전이 결정 (게임로직이 담당)
    // 반환: 새 액션명 | null (전이 없음)
    resolveAction(entityId, entity, actionType, actionTick, inputs, physicsState) {
      // physicsState: { x, y, vx, vy, onGround, facing }
      // 입력과 현재 상태를 보고 다음 액션 결정
      // 예: 점프 중 더블업 → 'BLOCK'
    },

    // 공이 바닥에 닿았을 때 (득점 판정)
    onBallHitFloor(state, side) {
      // side: 'left' | 'right'
      // state를 직접 변경하거나 변경 사항 객체 반환
    },

    // 공이 네트에 닿았을 때
    onBallHitNet(state) { },

    // 공이 플레이어에 닿았을 때
    onBallHitPlayer(state, entityId, hitNormal) { },
  },
});
```

## 5. 상태(State) 구조

GameLoop가 매 틱 갱신하고 Renderer가 읽는 전체 상태 객체.

```js
{
  // ─── 물리 상태 (physics unit, GameLoop이 관리) ─────────────
  [entityId]: {
    x, y,           // 물리 좌표 (원점 기준)
    vx, vy,         // 속도
    onGround,       // boolean (플레이어만)
    facing,         // 1 | -1 (플레이어만)
    actionType,     // 현재 액션 이름 (Renderer가 읽음)
    actionTick,     // 현재 액션 경과 틱 (Renderer가 spriteIndex 계산에 사용)
    actionDuration, // 현재 액션 총 틱 (Renderer가 spriteIndex 계산에 사용)
    stamina,        // 체력 (플레이어만)
  },

  // ─── 게임 규칙 상태 (handlers가 관리) ──────────────────────
  phase:      'serve' | 'rally' | 'point' | 'gameover',
  score:      { p1: number, p2: number },
  sets:       { p1: number, p2: number },
  serveOwner: 'p1' | 'p2',
  pointTimer: number,   // 득점 후 대기 틱
}
```

## 6. GameLoop.js 구현 내용

### 생성자

```js
new GameLoop({ entityManager, physicsMap, handlers })
```

### tick(state, inputs) 처리 순서

```
1. 각 플레이어 엔티티 처리
   a. actionTick++
   b. 액션 만료 체크 (actionTick >= actionDuration > 0)
      → handlers.resolveAction 호출하여 기본 액션 복귀
   c. 입력 기반 액션 전이
      → handlers.resolveAction 호출
      → 새 액션이면 actionType, actionTick=0, actionDuration 갱신
   d. 히트박스 갱신
      → entity.actions[actionType].getHitbox(t, facing) 호출
   e. 속도 갱신 (입력에 따른 vx, 중력에 따른 vy)
   f. 위치 갱신 (x += vx, y += vy)
   g. 바닥 충돌 (y <= 0 → onGround=true, vy=0, y=0)
   h. 좌우 맵 경계 clamp
   i. 코트 영역 clamp (playerSide 기준 네트 위치로 제한)

2. 공 처리
   a. vy에 중력 적용
   b. 위치 갱신
   c. 공 vs 맵 경계
      - 좌/우 벽: resolveCollision → 반사
      - 바닥: handlers.onBallHitFloor 호출 (득점 처리)
      - 상단: 통과 (충돌 없음)
   d. 공 vs 각 플레이어 히트박스
      → detectBallVsCompound
      → resolveCollision (공=탄성, 플레이어=비탄성)
      → handlers.onBallHitPlayer 호출
   e. 공 vs 네트
      → 충돌 해소, handlers.onBallHitNet 호출

3. Renderer용 상태 갱신
   - actionType, actionTick, actionDuration을 state에 기록
     (Renderer가 spriteIndex 계산에 사용)

4. { nextState, toPlay } 반환
   - toPlay는 handlers가 이벤트 처리 중 추가
```

## 7. Renderer.js 변경 내용

spriteIndex를 엔티티 설정과 상태에서 자동 계산.

```js
// 현재
const frame = asset[entity.spriteIndex ?? 0];

// 변경 후
const spriteIndex = computeSpriteIndex(entity, state);
const frame = asset[spriteIndex];

function computeSpriteIndex(entity, state) {
  const actionDef = entity.actions?.[state.actionType] ?? entity.actions?.DEFAULT;
  if (!actionDef?.sprites) return 0;
  const { start, count } = actionDef.sprites;
  if (count <= 1 || !state.actionDuration) return start;
  const t = state.actionTick / state.actionDuration;
  return start + Math.round((count - 1) * t);
}
```

## 8. StateSystem.js 변경 내용

tick 콜백 위임 로직 제거. 순수 상태 컨테이너만 유지.

```js
// 변경 전: initState/tick 콜백 주입
constructor(match, entityManager, { initState, tick }) { ... }
tick(inputs) { ... } // 콜백 호출

// 변경 후: 초기 상태만 받음
constructor(initialState) { this.#state = initialState; }
setState(nextState) { this.#state = nextState; }
get buf() { return this.#state; }
```

## 9. GameBuilder.js 변경 내용

`start(match, logic)` → `start(config)` 인터페이스 변경.

```js
// 변경 전
game.start(match, { initState, tick });

// 변경 후
game.start({
  physicsMap,
  initEntities,
  handlers,
});
```

내부 Game 클래스:
- `initEntities(entityManager)` 호출하여 초기 상태 획득
- `GameLoop` 인스턴스 생성 (entityManager, physicsMap, handlers 주입)
- RAF 루프에서 `gameLoop.tick(state, inputs)` 호출

## 11. 구현 순서

1. **EntityManager** — `actions` 포함 등록 지원, `getEntity(id)` 활용 확인
2. **StateSystem** — 순수 컨테이너로 축소
3. **GameLoop.js** — 핵심 tick 구현 (액션 상태머신 → 물리 → 이벤트)
4. **GameBuilder.js** — `start(config)` 인터페이스로 변경, GameLoop 소유
5. **Renderer.js** — `computeSpriteIndex` 추가
6. **game/GameLogic.js** — handlers 구현 (액션 전이, 득점 처리 등)
7. **game/characters/** — 캐릭터별 설정 파일 작성

## 12. 미결 사항

| 항목 | 내용 |
|------|------|
| 서브 페이즈 처리 | 나중에 구현 |
| 체력 소모 계산 | 나중에 구현. 캐릭터 스탯별로 달라짐 |

### 확정 사항
- **렌더 좌표 변환**: Renderer가 담당. state에는 physics 좌표만 저장. 픽셀은 Renderer만 앎.
- **네트**: 엔티티로 EntityManager에 등록.
